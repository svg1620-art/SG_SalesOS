# CLAUDE.md — SG_SalesOS

Операционная система управления качеством продаж для ServiceGuru: ОКК + коучинг менеджеров + ежедневный агент-сводка для РОПа. Полное ТЗ — в `SG_SalesOS_spec.md`. Этот файл читается каждой сессией Claude Code.

## Миссия
Звонок отдела продаж → авто-транскрибация с диаризацией → оценка по чек-листу (балл + зона 🟢/🟡/🔴) → рекомендации по навыкам + упущенные моменты. РОП видит всё на одном экране, менеджер — только себя. Раз в день — AI-сводка «на что обратить внимание».

## Стек (не менять без причины)
- **Backend:** Python 3.11+, Flask + Jinja2 + HTMX. Без React.
- **DB:** PostgreSQL + SQLAlchemy. Миграции — Flask-Migrate (Alembic).
- **Фон:** APScheduler (обработка звонков + дневная сводка).
- **Графики:** Chart.js (donut зон, линии трендов, радар по критериям).
- **ИИ анализ/сводка:** Anthropic Claude (модель из env, не хардкодить).
- **Транскрибация:** OpenAI `gpt-4o-transcribe`, диаризация через split стерео-каналов (`pydub`/`ffmpeg`).
- **Auth:** Flask-Login, две роли: `admin`, `manager`.
- **Деплой:** Railway (регион **EU**), Volume на `/data`.

## Жёсткие правила окружения (Railway) — усвоено на практике
- Регион проекта — **EU**, иначе нет доступа к Anthropic API с РФ-инфры. Не деплоить в US.
- Переменные окружения добавлять **по одной** через «New Variable». Копипаст блоком может незаметно склеить значения. **Без кавычек** вокруг значений.
- `DATABASE_URL` — следить за кириллицей (был баг с невидимыми символами).
- Файловая система эфемерна между деплоями. **Всё, что должно пережить рестарт (аудио), — только в Volume `/data`.** Никаких файловых сессий/кэшей на локальном диске.

## Правила кода
- **Никаких хардкод-секретов и моделей.** Всё через `config.py` из env: `DATABASE_URL`, `SECRET_KEY`, `ANTHROPIC_API_KEY`, `OPENAI_API_KEY`, `CLAUDE_MODEL`, `CLAUDE_MODEL_DIGEST`, `OPENAI_TRANSCRIBE_MODEL`, `AUDIO_DIR` (=/data), `POLL_INTERVAL_MIN`, `DIGEST_HOUR`, `TZ`, amoCRM-переменные.
- **Парсинг JSON от Claude — через извлечение с балансировкой скобок**, не наивный `json.loads` (модель может обернуть в ```json). Единая утилита `extract_json()`.
- **Пиннинг:** `openai>=1.57.4`, `httpx==0.27.2` (связка `openai==1.51.2`+`httpx==0.28.0` крашится). Anthropic SDK — свежий.
- **Нормализация телефона** к `+7XXXXXXXXXX` — единая функция `normalize_phone()`. По ней матчим `Client` и агрегируем `Dialog`.
- **Изоляция сбоев по источнику/звонку:** падение одного звонка не роняет весь фоновый прогон; ошибка пишется в `Call.error`, статус `failed`.
- **Ingestion абстрагирован:** интерфейс `CallSource` → `AmoCallSource` (основной) и `MegafonCallSource` (запасной: вебхук `history` + FTP). Пайплайн обработки один.
- Комментарии и UI — на русском.

## Диаризация (проверено на реальной записи Мегафона)
Записи Мегафон ВАТС — честное стерео, стороны разговора на разных каналах (проверено: L/R независимы). Режем на два моно, транскрибируем отдельно, склеиваем по таймингам → «Менеджер / Клиент». Какой канал = менеджер — флаг `manager_channel` на аккаунт (для входящих обычно определяется по направлению). Фолбэк для моно: разметка спикеров Claude по смыслу реплик (`diarization='heuristic'`).

## Оценка и зоны
- Каждый критерий чек-листа Claude оценивает 0–10. Итог = Σ(score/10 × weight), веса в сумме 100.
- Пороги (дефолт, редактируются на чек-листе): 🟢 ≥80, 🟡 60–79, 🔴 <60.
- **Правило критичных критериев:** если критичный критерий (`is_critical`) получил 0/10 — зона не может быть 🟢 (капается в 🟡), даже если сумма ≥80.
- Зона считается **на звонок**. Диалог хранит агрегат (avg_score, last_zone, trend).
- Дефолтный чек-лист (8 критериев, домен-агностичный) — Приложение A в ТЗ, заливается сидом на Этапе 2.

## Модель данных (кратко; детали в ТЗ, разд. 4)
`User(role)`, `Checklist`, `Criterion(weight,is_critical)`, `Client(phone_normalized)`, `Dialog`, `Call(status,transcript_json,overall_score,zone)`, `CallCriterionScore`, `Recommendation`, `MissedMoment`, `DailyDigest`, `AmoToken`.
Статусы звонка: `new→downloading→transcribing→analyzing→done` / `failed(+error)`.
Дедуп: `amo_note_id` unique; ручная загрузка — SHA256(имя+длительность+дата).

## Этапы сборки (по одной сессии, полные файлы, деплой+проверка после каждой)
1. Каркас: фабрика, config, все модели, миграции, auth (admin/manager), layout SG, сид админа из env. Deploy Railway EU + Postgres + Volume.
2. Чек-листы: CRUD + критерии/веса/критичность/пороги + активация; сид дефолт-чек-листа; AI-генерация чек-листа из описания.
3. Ручная загрузка + пайплайн: транскрибация(стерео)→анализ Claude→баллы/зона/саммери/рекомендации/упущения; фоновый worker + polling-статус.
4. Карточка звонка: плеер, транскрибация с ролями + инлайн-подсветка упущений, саммери, скоркарта (радар+цитаты), рекомендации.
5. Агрегация диалогов по номеру телефона.
6. Дашборд РОПа: KPI, donut зон, лидерборд, лента с фильтрами.
7. Кабинет менеджера: свои звонки, тренд, рекомендации.
8. Интеграция amoCRM: OAuth, опрос notes (call_in/call_out), скачивание записей в Volume, авто-привязка, APScheduler.
9. Агент дневной сводки: cron Claude → блок на дашборде.

**Definition of done этапа:** код деплоится на Railway без ошибок, миграции применяются, заявленный функционал проверяется вручную, ничего из прошлых этапов не сломано.
