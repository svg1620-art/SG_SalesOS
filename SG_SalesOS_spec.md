# SG_SalesOS — Техническое задание

**Операционная система управления качеством продаж**
ОКК + рекомендации менеджерам + ежедневный агент-сводка для РОПа

Версия ТЗ: 1.0
Заказчик: ServiceGuru (внутренний продукт)
Автор: Serg

---

## 1. Суть продукта

Один продукт закрывает три задачи поверх звонков отдела продаж:

1. **ОКК (отдел контроля качества).** Каждый звонок автоматически транскрибируется, разбирается по чек-листу и получает балл + зону (🟢 / 🟡 / 🔴).
2. **Коучинг менеджеров.** Менеджер видит по своим звонкам рекомендации по навыкам и подсвеченные упущенные моменты прямо в транскрибации.
3. **Агент-сводка для РОПа.** Раз в день формируется сводка «на что обратить внимание»: кто просел, какие диалоги в красной зоне, тренды.

Главный экран РОПа: количество диалогов с клиентами, количество звонков, распределение по зонам, лидерборд менеджеров, лента звонков с фильтрами и провалом в любой звонок (транскрибация + саммери + анализ + плеер).

## 2. Границы и ключевые решения

| Параметр | Решение |
|---|---|
| Тип | Внутренний продукт ServiceGuru (single-tenant, один аккаунт). Мультиарендность не делаем, но структуру ролей закладываем. |
| Домен | **Домен-агностично.** Чек-листы создаются админом под любую сферу (HoReCa, медицина и т.д.). Никакой хардкод-специфики HoReCa. |
| Роли | **Администратор** (РОП/ОКК — видит всех, управляет чек-листами) и **Менеджер** (видит только свои звонки и рекомендации). |
| Источник звонков | Автоматическая выгрузка из **amoCRM** (телефония — Мегафон, пишет записи в amoCRM). На старте — ручная загрузка аудио. |
| Привязка звонка | Менеджер и клиент берутся из данных amoCRM (`responsible_user_id`, телефон контакта). |
| «Диалог» | Агрегат звонков по нормализованному **номеру телефона** клиента. |
| Зоны | Балл по чек-листу. Зона считается **на звонок**. Диалог показывает последнюю/агрегированную зону. |
| Рекомендации | **Коучинг по навыкам** (не next-step по сделке). Упущенные моменты — инлайн в транскрибации + сводным списком. |
| Дневная сводка | В интерфейсе (email/Telegram — на будущее). |
| Стек | Flask + Jinja2 + HTMX + PostgreSQL + Chart.js. Транскрибация — OpenAI `gpt-4o-transcribe`. Анализ/сводка — Claude Sonnet. |
| Масштаб | ~4 менеджера, ~30 звонков/день (~900/мес). |
| Деплой | **Railway EU region** (обязательно для доступа к Anthropic API с российской инфраструктуры), GitHub web UI, Railway Volume для аудио. |

## 3. Архитектура

```
amoCRM (notes: call_in/call_out) ──poll (APScheduler)──▶ ingest
                                                            │
                          ручная загрузка аудио ────────────┤
                                                            ▼
                                              [1] скачать запись → Volume (/data)
                                                            ▼
                                   [2] транскрибация (OpenAI gpt-4o-transcribe)
                                        диаризация: split стерео-каналов
                                                            ▼
                                   [3] анализ (Claude Sonnet, активный чек-лист)
                                        → баллы по критериям, зона, саммери,
                                          рекомендации, упущенные моменты
                                                            ▼
                                   [4] агрегация диалога по номеру телефона
                                                            ▼
                        PostgreSQL ◀──────────────────────────┘
                             │
          ┌──────────────────┼───────────────────────┐
          ▼                  ▼                        ▼
   Дашборд РОПа        Кабинет менеджера        Агент дневной сводки
   (HTMX + Chart.js)   (свои звонки)            (Claude Sonnet, cron 1/день)
```

### Пайплайн обработки звонка (статусы)
`new` → `downloading` → `transcribing` → `analyzing` → `done` (или `failed` с сохранением ошибки).
Обработка идёт в фоне (APScheduler worker), UI показывает статус polling'ом (как в SG_MenuParser).

### Диаризация
> ✅ Проверено на реальной записи Мегафона (`..._in_..._2026_07_02...mp3`): файл — **честное стерео**, стороны разговора разведены по каналам (L/R независимы, разница L−R ≈ уровню каналов). 16 кГц, mp3. Стерео-диаризация работает без эвристики.

- Если запись стерео (2 канала — стандарт Мегафон ВАТС: одна сторона на канал) — режем через `pydub`/`ffmpeg` на два моно-файла, транскрибируем отдельно, склеиваем по таймингам → чистое разделение «Менеджер / Клиент».
- Если моно — один прогон, спикеров размечает Claude на этапе анализа по смыслу реплик (фолбэк, помечается как `diarization: heuristic`).
- Какой канал = менеджер, определяется по направлению звонка (`call_out` → менеджер инициатор) + настройка на аккаунт.

## 4. Модель данных (PostgreSQL)

```
User
  id, email, password_hash, full_name, role (admin|manager),
  amo_user_id (nullable, для сопоставления с ответственным в amoCRM),
  is_active, created_at

Checklist
  id, name, description, domain (текст, напр. "HoReCa", свободный),
  zone_green_min (по умолчанию 80), zone_yellow_min (по умолчанию 60),
  is_active (bool), created_at
  # зона: score >= green_min → 🟢; >= yellow_min → 🟡; иначе → 🔴
  # + правило: если провален (0 из 10) любой критичный критерий →
  #   зона не может быть 🟢 (капается в 🟡 максимум)

Criterion
  id, checklist_id (FK), title, description (что считается "хорошо"),
  weight (int, вклад в итог), order_index, is_critical (bool)

Client
  id, phone_normalized (unique), name (из amoCRM), amo_contact_id (nullable),
  first_seen_at, last_seen_at

Dialog                       # агрегат звонков по клиенту
  id, client_id (FK), manager_id (FK User), calls_count,
  avg_score, last_zone, trend (up|down|flat), updated_at

Call
  id, dialog_id (FK), manager_id (FK User), client_id (FK),
  checklist_id (FK, каким чек-листом оценён),
  amo_note_id (nullable, unique), direction (in|out),
  started_at, duration_sec,
  source_link (ссылка amoCRM/телефонии),
  audio_path (путь в Volume),
  status (new|downloading|transcribing|analyzing|done|failed),
  error (nullable),
  transcript_json (реплики с спикером и таймингами),
  summary (текст),
  overall_score (int), zone (green|yellow|red),
  diarization (stereo|heuristic),
  created_at, processed_at

CallCriterionScore           # разбалловка по критериям
  id, call_id (FK), criterion_id (FK),
  score (int), max_score (int),
  evidence (цитата из транскрибации), comment,
  is_missed (bool)           # критерий провален/упущен

Recommendation               # коучинг по навыкам
  id, call_id (FK), manager_id (FK),
  skill (тег навыка, напр. "Выявление потребностей"),
  text (рекомендация), priority (high|med|low)

MissedMoment                 # упущенные моменты для инлайн-подсветки
  id, call_id (FK), transcript_span_start, transcript_span_end,
  label, explanation

DailyDigest                  # дневная сводка РОПа
  id, date (unique), content_json, created_at

AmoToken                     # OAuth-токены amoCRM (единственная строка)
  id, access_token, refresh_token, expires_at, base_domain
```

**Правило нормализации телефона:** приводим к `+7XXXXXXXXXX` (убираем пробелы/скобки/дефисы, `8` → `+7`). По нему матчим `Client` и агрегируем `Dialog`.

**Дедупликация звонков:** по `amo_note_id` (unique). Для ручной загрузки — по SHA256(нормализованное имя файла + длительность + дата).

## 5. Обработка ИИ

### 5.1 Транскрибация (OpenAI)
- Модель: `gpt-4o-transcribe`, язык `ru`.
- Пиннинг версий (важно, известная связка-краш): `openai>=1.57.4`, `httpx==0.27.2`.
- Стерео → два прохода по каналам → merge по времени.
- Результат в `transcript_json`: массив `{speaker: "manager"|"client", start, end, text}`.

### 5.2 Анализ звонка (Claude Sonnet)
Один структурированный вызов. В промпт подаём: транскрибацию с ролями + активный чек-лист (критерии, описания, веса, пороги). Просим строго JSON.

Ответ:
```json
{
  "overall_score": 0-100,
  "summary": "3-5 предложений",
  "criteria": [
    {"criterion_id": 12, "score": 8, "max_score": 10,
     "evidence": "цитата", "comment": "...", "is_missed": false}
  ],
  "recommendations": [
    {"skill": "Отработка возражений", "priority": "high",
     "text": "конкретный совет менеджеру"}
  ],
  "missed_moments": [
    {"span_start": 142, "span_end": 210,
     "label": "Не выявлен бюджет",
     "explanation": "клиент дал сигнал, менеджер не уточнил"}
  ]
}
```
- **Парсинг JSON — через извлечение с балансировкой скобок**, не наивный `json.loads` (Claude может обернуть в ```json). Переиспользовать проверенную функцию-экстрактор.
- Зона считается из `overall_score` и порогов активного чек-листа.
- `overall_score` = взвешенная сумма критериев (Claude считает, бэкенд валидирует пересчётом по весам).

### 5.3 Дневная сводка (Claude Sonnet, cron)
Раз в день (напр. 20:00) агрегирует все звонки за день:
- кто из менеджеров просел (падение среднего балла),
- список звонков в красной зоне (со ссылками),
- частые упущенные моменты по отделу,
- 2-3 фокуса для РОПа на завтра.
Сохраняется в `DailyDigest`, показывается блоком на дашборде.

## 6. Интерфейсы

### 6.1 Дашборд РОПа (главный экран)
- **KPI-карточки:** диалогов с клиентами, звонков за период, распределение зон (🟢/🟡/🔴 — числа и %).
- **Donut** распределения зон (Chart.js).
- **Лидерборд менеджеров:** средний балл, распределение зон, кол-во звонков, стрелка тренда.
- **Блок «Сводка на сегодня»** (агент).
- **Лента звонков/диалогов:** фильтры по зоне / менеджеру / дате, красная зона подсвечена. Клик → карточка звонка.

### 6.2 Карточка звонка (провал в звонок)
- Аудиоплеер (из Volume).
- **Транскрибация** с ролями (Менеджер/Клиент) и **инлайн-подсветкой упущенных моментов** (наведение → пояснение).
- **Саммери**.
- **Скоркарта по критериям** (радар Chart.js + список с цитатами-доказательствами).
- **Рекомендации** по навыкам.

### 6.3 Кабинет менеджера
- Только свои звонки. Свой тренд балла.
- Агрегированные рекомендации + по каждому звонку.
- Упущенные моменты. Блок «что улучшить на этой неделе».

### 6.4 Управление чек-листами (админ)
- CRUD чек-листов, критериев (вес, описание, критичность), пороги зон, активация.
- Кнопка **«Сгенерировать чек-лист из описания»** (Claude драфтит критерии по тексту процесса продаж).

### 6.5 Дизайн
Бренд ServiceGuru: тёмная база `#0A0A0D`, синий `#1467F5`, циан `#00BFDC`; Manrope (дисплей), JetBrains Mono (цифры). Логотипы — `tech.service.guru/style/brandbook/`.

## 7. Интеграция amoCRM (Этап 8)

- **Авторизация:** OAuth2, долгоживущий refresh-токен с авто-обновлением. Токены в `AmoToken`.
- **Получение звонков:** опрос примечаний `/api/v4/{entity}/notes` с `note_type=call_in|call_out`, фильтр по `updated_at` с последней синхронизации. Из `params`: `LINK` (mp3), `PHONE`, `DURATION`; из примечания — `responsible_user_id` (→ менеджер через `User.amo_user_id`), сущность-контакт (→ клиент).
- **Скачивание записи:** сразу тянем mp3 по `LINK` в Volume `/data`. Мегафон хранит записи на сервере **1 год**, но кэшируем к себе всё равно — на случай удаления контакта в amoCRM (тогда пропадают и звонки) и для независимости плеера от внешних ссылок.
- **Расписание:** APScheduler, `POLL_INTERVAL_MIN` (по умолчанию 15).
- Ручная загрузка остаётся как фолбэк.

### 7.1 Запасной путь — напрямую от Мегафон ВАТС (заложить, включить при необходимости)
Мегафон ВАТС даёт свою интеграцию, минуя amoCRM:
- **REST API (вебхук).** ВАТС шлёт POST на наш endpoint по событиям: `event` (звонок начался/направление/номера), `history` (звонок завершён — длительность, статус, **ссылка на запись**). Точные поля — в доке, которую качаем из панели ВАТС: **Настройки → Интеграция с CRM → Ваша CRM / REST API**. Это ниже задержка, чем поллинг amoCRM.
- **FTP-бэкфилл.** Все записи доступны по FTP под учёткой администратора ВАТС (`логин@company.megapbx.ru`, пассивный режим). Годится для первичной массовой загрузки истории.
- **Имя файла записи** несёт метаданные: `<номер_А>_<in|out>_<номер_Б>_<YYYY_MM_DD-HH_MM_SS>.mp3` → направление, оба номера, время. Для входящего (`in`) A — клиент, B — наш номер.

Архитектурно ingestion абстрагируем: интерфейс `CallSource` с реализациями `AmoCallSource` (основная) и `MegafonCallSource` (вебхук+FTP). Пайплайн обработки один и тот же.

## 10. Поэтапный план сборки (Claude Code, 9 сессий)

Каждый этап = отдельная сессия, полные замены файлов (не диффы). После каждого — деплой на Railway и проверка.

**Этап 1 — Каркас.**
Flask-фабрика, `config`, все модели, миграции, аутентификация (роли admin/manager), базовый layout в бренде SG, сид админа. Деплой на Railway (EU), Postgres, Volume `/data`. Проверка: логин, пустой дашборд.

**Этап 2 — Чек-листы.**
CRUD чек-листов и критериев (веса, описания, критичность), пороги зон, активация. AI-генерация чек-листа из описания процесса (Claude). Проверка: создан и активирован чек-лист.

**Этап 3 — Ручная загрузка + пайплайн (ядро ценности).**
Форма загрузки аудио + ручная привязка менеджера/клиента. Пайплайн: транскрибация (стерео-диаризация) → анализ Claude по активному чек-листу → баллы, зона, саммери, рекомендации, упущенные моменты. Фоновый worker + polling-статус. Проверка: реальный звонок проходит до `done`.

**Этап 4 — Карточка звонка.**
Плеер, транскрибация с ролями и инлайн-подсветкой упущенных моментов, саммери, скоркарта (радар + цитаты), рекомендации. Проверка: «провалиться в звонок» полностью работает.

**Этап 5 — Агрегация диалогов.**
Нормализация телефона, склейка звонков в диалог по номеру, пересчёт avg_score/last_zone/trend, экран диалога. Проверка: несколько звонков одного клиента = один диалог.

**Этап 6 — Дашборд РОПа.**
KPI-карточки, donut зон, лидерборд менеджеров, лента с фильтрами (зона/менеджер/дата), подсветка красной зоны. Проверка: главный экран сходится с данными.

**Этап 7 — Кабинет менеджера.**
Свои звонки, тренд балла, агрегированные рекомендации, упущенные моменты, «что улучшить на неделе». Проверка: менеджер видит только себя.

**Этап 8 — Интеграция amoCRM.**
OAuth, опрос notes (`call_in/call_out`), скачивание записей в Volume, авто-привязка менеджера/клиента, APScheduler. Ручная загрузка как фолбэк. Проверка: звонки из amoCRM появляются и обрабатываются сами.

**Этап 9 — Агент дневной сводки.**

## 12. Пороги зон (финал)

По умолчанию для нового чек-листа:
- 🟢 **Зелёная** — score ≥ **80** (сильный звонок)
- 🟡 **Жёлтая** — **60–79** (приемлемо, есть пробелы → внимание)
- 🔴 **Красная** — **< 60** (слабый звонок → разбор обязателен)

Пороги редактируются на уровне чек-листа. Правило критичных критериев: если хоть один критичный критерий получил 0/10, звонок не может быть 🟢 (максимум 🟡), даже если сумма ≥ 80 — чтобы «формально набрал балл, но не выявил потребность / не закрыл на следующий шаг» не попадало в зелёную зону.

## Приложение A. Дефолтный чек-лист «Универсальная оценка звонка отдела продаж»

Домен-агностичный, применим к любой сфере. Веса в сумме = 100. Каждый критерий Claude оценивает 0–10, итог = Σ(score/10 × weight). Критичные помечены ⚠️.

| # | Критерий | Вес | Крит. | Что считается «хорошо» |
|---|---|---|---|---|
| 1 | Установление контакта | 8 | | Поздоровался, представился, назвал компанию, доброжелательный тон, обозначил цель звонка. |
| 2 | Выявление потребностей / квалификация | 20 | ⚠️ | Задавал открытые вопросы, выяснил задачу и боль клиента, ситуацию, критерии выбора, кто принимает решение, сроки/бюджет (по применимости). |
| 3 | Активное слушание | 10 | | Не перебивал, уточнял, резюмировал слова клиента, реагировал на сказанное, а не по скрипту. |
| 4 | Презентация через ценность | 18 | ⚠️ | Говорил на языке выгод под выявленную потребность (а не перечислял функции), связывал решение с болью клиента. |
| 5 | Работа с возражениями | 15 | ⚠️ | Выслушал возражение, не спорил, отработал по существу, проверил, что возражение снято. |
| 6 | Инициатива и ведение диалога | 8 | | Вёл разговор, удерживал структуру, не отдавал инициативу клиенту, управлял темпом. |
| 7 | Договорённость о следующем шаге | 15 | ⚠️ | Зафиксировал конкретный следующий шаг с датой/временем (встреча, КП, повторный звонок), получил согласие. |
| 8 | Речь и вежливость | 6 | | Грамотная чистая речь, без слов-паразитов и грубости, корректное завершение разговора. |

**Критичные (⚠️):** 2, 4, 5, 7 — ядро продажи. Их провал (0/10) блокирует зелёную зону.

Этот чек-лист заливается сидом на Этапе 2 как активный по умолчанию. Админ правит веса/описания или создаёт отдельные чек-листы под другие сферы (медицина и т.д.), в т.ч. через AI-генерацию из описания процесса.
